### C. 오늘의 문제 1개
- 문제/증상:
  - 버튼 스타일이 화면마다 늘어나면 **디자인 일관성이 깨지고**, 이후 수정 시 버튼을 여러 군데 고쳐야 하는 유지보수 비용이 커질 수 있음
- 원인 가설:
  - 버튼 요구사항이 (가로/세로) + (모서리 3종) + (아이콘 위치/조합) + (사이즈 3종)으로 늘어나는데, 규격 없이 만들면 중복 컴포넌트가 생김
  - 색상/아이콘 색상 처리를 컴포넌트 내부에서 흩어지게 하면 variant 추가 시 수정 범위가 커짐
  - 케이스가 많아도 “빠르게 확인”하는 화면이 없으면 UI 깨짐을 늦게 발견함
- 확인 방법:
  - 버튼 가이드의 케이스(아이콘만/아이콘+텍스트 조합, rounded/square/pill, size sm/md/lg)가 실제 컴포넌트 props로 전부 표현되는지 확인
  - 테스트 페이지에서 모든 조합을 렌더링해 정렬/간격/호버 동작을 한 번에 점검
- 해결:
  - 버튼을 2개 컴포넌트(Parallel/Vertical)로 나누고, 나머지는 **props(variant/size/radius/iconPosition 등)**로 통합
  - 색상/아이콘 필터를 상수로 외부 export해 재사용 및 확장(variant 추가)에 대비
  - 테스트 페이지에서 조합 케이스를 자동으로 돌려보도록 구성
- 결과:
  - 버튼 디자인 규격을 코드로 고정해 **화면 전체 일관성을 유지할 기반**이 생김
  - 신규 variant/아이콘 조합 추가 시 “상수 추가 + props 적용” 수준으로 확장 가능해짐
  - 조합 테스트 페이지가 있어 UI 회귀(깨짐)를 빠르게 잡을 수 있음
- 다음 액션:
  - hover를 JS 이벤트가 아니라 Tailwind/CSS(예: group-hover)로 전환해 스타일 로직 분리
  - disabled/loading/focus(접근성) 상태 정의 및 적용(aria-label, 키보드 포커스 링)
  - 아이콘 경로를 문자열로 넘기는 부분을 실제 import 방식으로 통일(번들링/경로 안정화)

- 한 줄 설명:
