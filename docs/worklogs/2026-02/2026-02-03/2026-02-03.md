### B. 오늘 한 일(사실만)
- 작업 1: Button 컴포넌트 타입을 2종으로 정의
  - parallelButton(가로형 기본)
  - verticalButton(세로형 기본)
- 작업 2: 버튼 모양 variant를 3종으로 정리
  - rounded(둥근 모서리) / square(각진 모서리) / pill(완전히 둥근)
- 작업 3: 버튼 콘텐츠 구성을 케이스별로 정리
  - parallelButton: (1) 기본 텍스트, (2) 아이콘만, (3) 왼쪽 아이콘+오른쪽 텍스트, (4) 왼쪽 텍스트+오른쪽 아이콘
  - verticalButton: (1) 아이콘만, (2) 위 아이콘+아래 텍스트, (3) 위 텍스트+아래 아이콘
  - size: sm(14px) / md(16px) / lg(18px)

### C. 오늘의 문제 1개
- 문제/증상:
  - 화면마다 버튼 형태가 달라지면 스타일이 쉽게 분산되고, 추후 변경 시 버튼을 여러 군데 수정해야 하는 문제가 생길 수 있음
- 원인 가설:
  - “버튼 종류/사이즈/아이콘 위치” 기준이 없으면 페이지별로 서로 다른 구현이 생김
  - 모서리 스타일(rounded/square/pill)과 레이아웃(parallel/vertical)이 섞이면 케이스가 늘어나 관리가 어려워짐
  - 텍스트+아이콘 조합을 props로 통일하지 않으면 컴포넌트가 쪼개져 재사용성이 떨어짐
- 확인 방법:
  - 버튼 요구사항을 “레이아웃(가로/세로) × 모서리(3종) × 콘텐츠(아이콘/텍스트 조합) × size(3종)”으로 분해해 조합이 커버되는지 점검
  - 실제 UI에서 등장하는 버튼(예: ‘품질 검수’, ‘이전’)이 정의된 케이스로 표현 가능한지 확인
- 해결:
  - Button을 단일 컴포넌트로 두고, props로 레이아웃/모서리/아이콘 위치/size를 제어하는 방향으로 규격을 확정
  - parallel/vertical 각각에 대해 “아이콘만/텍스트+아이콘 조합”을 표준 케이스로 정리
- 결과:
  - 버튼 스타일/사용 규칙이 명확해져 이후 페이지 구현 시 **버튼 UI 일관성**을 유지할 수 있는 기준이 생김
  - 사이즈(sm/md/lg) 기준이 고정되어 디자인-개발 간 협업 시 기준점이 생김
- 다음 액션:
  - Button props 스펙 확정(예: `layout`, `shape`, `size`, `icon`, `iconPosition`, `children`, `disabled`)
  - 상태 디자인 정의(hover/active/disabled/loading)와 접근성(키보드 포커스, aria-label) 체크
  - 실제 페이지(Upload/Attribute/Diagram)에 적용하면서 추가 케이스가 필요한지 검증
