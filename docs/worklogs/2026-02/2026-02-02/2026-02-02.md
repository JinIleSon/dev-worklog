### C. 오늘의 문제 1개
- 문제/증상:
  - 화면이 여러 개(업로드/데이터정보/속성검사/공간검사)로 확장되는데, 컴포넌트를 페이지별로 즉흥적으로 만들면 **중복이 폭증하고 유지보수가 어려워질 위험**이 있음
- 원인 가설:
  - 공통 컴포넌트의 경계(“진짜 공통” vs “한 페이지 전용”)가 안 잡히면 같은 UI를 여러 번 만들게 됨
  - 카드/결과/검사 UI가 종류가 많아 variant 정책 없이 가면 폴더가 난잡해짐
  - import 경로가 길어지면 사용처가 복잡해지고 리팩토링 비용이 커짐
- 확인 방법(학습/설계 기준):
  - 설계(PDF)에 등장하는 UI 요소를 전부 나열해서 “공통 컴포넌트로 뺄 것”과 “페이지 전용”을 분리
  - Card 계열을 ‘업로드/파일명 규칙/검사 대상/검사 결과/검사 선택’으로 묶어 variant 설계가 가능한지 점검
- 해결:
  - `components/common`을 중심으로 재사용 범위를 먼저 결정하고, Page는 조립(Composition)만 담당하도록 구조화
  - Card 하위 구조를 목적 기준으로 분리:
    - Upload(업로드 카드 3종)
    - Rule(파일명 규칙 2종)
    - Target(검사 대상 1종)
    - Result(AttributeResult/DiagramResult - 축약+상세)
    - Check(AttributeCheck/DiagramCheck - 선택 O/X)
  - `index.js`를 도입해 import 경로를 단순화(폴더/파일명이 겹치는 케이스 포함)
- 결과:
  - 구현 전에 “공통 컴포넌트/레이아웃/페이지” 책임이 분리되어, 이후 화면 확장 시에도 **중복 없이 쌓을 수 있는 뼈대**가 만들어짐
  - UI 설계에서 요구하는 핵심 컴포넌트(버튼, 카드, 테이블, 셀렉트, 레이아웃, 오류 모달, 폰트)를 기준으로 구조가 정렬됨
- 다음 액션:
  - 공통 컴포넌트별 props/variant 규칙 문서화(예: Button size/variant, Card 타입 enum)
  - Result/Check 카드의 “축약/상세(모달)” 데이터 구조(검사 항목, 합격/불합격, 오류코드 등) 타입 정의
  - 전역 스타일/타이포(폰트) 적용 방식 확정 및 pages에 적용 시작
