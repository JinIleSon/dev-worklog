### B. 오늘 한 일(사실만)

2) **데이터 위치 보기(지도) 화면 설계**

- 지도 도구(확대/축소/측정/속성조회 등) 배치
- 하단 입력 영역: **지역 선택 + 파일 목록 + 분류(대/중/소) 입력** 섹션 구성

3) **품질검사 결과 패널 UI 설계**
- **속성 정보 품질 검사**: 시간/결과/내용 기반 로그 리스트 UI 구성
- **지하 시설물 공간 정보 품질 검사**: 분류별(상수/하수/전력/가스/통신 등) 결과 테이블 구성
- 검사 항목(체크박스) + 오차범위 입력(버텍스/맨홀/맨홀 등) UI 정의
- 하단에 “검사 상세 로그” 영역 분리

### C. 오늘의 문제 1개
- **문제/증상**
  - 품질검사 결과가 **불합격**으로 반복 노출
  - 로그에 **오류코드 315**, `pipe_lm_source가 필요합니다` 형태 메시지 확인

- **원인 가설**
  - 업로드 데이터(DBF 속성 스키마)에 **필수 필드(pipe_lm_source)** 가 없거나
  - 필드 매핑/검증이 UI에서 선제적으로 걸러지지 않아 **검사 로직 단계에서 실패**한 것으로 보임 *(추측)*

- **확인 방법**
  - DBF 필드 목록(스키마)에서 `pipe_lm_source` 존재 여부 확인
  - 품질검사 기준 문서/룰에서 **필수 속성 목록 + 코드 315 정의** 확인
  - 동일 데이터로 재검사해 **재현성** 확인

- **해결**
  - 업로드 직후 **필수 속성 사전검증 UI/플로우**를 추가하는 방향으로 설계
  - 오류 로그 영역을 “무엇이 누락됐는지/어떻게 조치하는지”가 보이도록 컴포넌트 구조 정리

- **결과**
  - “관리자 문의” 수준에서 끝나지 않고, **사용자가 즉시 원인/조치(필수 필드 누락)** 를 파악할 수 있는 UX 방향을 확정

- **다음 액션**
  - 에러코드(예: 315)별 **원인/조치 문구 템플릿** 정의
  - 필수 필드 체크(사전검증) 규칙 확정 후 UI 반영
  - 로그 포맷을 **원인 / 조치 / 참고** 형태로 표준화
