### C. 오늘의 문제 1개
- **문제/증상:** 요구사항 문서가 텍스트 중심이라, 화면 설계/개발 단계에서 **“어느 화면의 어느 요소를 말하는지”** 해석 비용이 큼

- **원인 가설**
  - 요구사항이 기능/정책/예외가 섞여 있고, UI 요소와 1:1로 매핑되지 않아 누락이 발생하기 쉬움
  - 검수 관점에서는 “충족 여부”를 빠르게 판단해야 하는데 문서만으로는 시각적 근거가 약함

- **확인 방법**
  - 피그마 상에서 실제 화면 플로우 기준으로 요구사항을 화면 단위로 재배치했을 때 누락되는 항목이 있는지 확인
  - 빨간 표시(추가 필요) 항목을 따로 모아 “개발 TODO”로 추출 가능한지 확인

- **해결**
  - 요구사항을 피그마 화면 구조에 직접 이어붙이고(매핑) → 선/라벨링 → 누락은 빨간색 표시로 시각화
  - 결과물을 이미지/PDF로 내보내 팀 내 공유 및 리뷰(검수) 가능한 형태로 정리

- **결과**
  - 텍스트 요구사항이 “화면 단위 체크리스트”로 변환되어, 리뷰 시 충족/미충족 판단 속도가 올라감
  - 빨간 표시 항목이 곧바로 “추가 구현 리스트”로 남아 다음 액션이 명확해짐

- **다음 액션**
  - 빨간 표시 항목을 개발 이슈/티켓(TODO)로 분리하고 우선순위(필수/권장) 태깅
  - 화면별 요구사항을 “수용 기준(AC: Acceptance Criteria)” 형태로 1~2줄씩 덧붙여 검수 기준을 고정
  - (가능하면) 실제 데이터 흐름(업로드→검사→결과)과 연결되는 API/데이터 스키마 초안도 화면 옆에 함께 표기
